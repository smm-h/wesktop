# App update, client cache, and version identity must be framework-owned

Companion to `desktop-runtime-wiring-belongs-in-the-framework.md` (that todo covers backend-URL injection, single-instance, process lifecycle, dev-mode port/output, and version skew). This one covers the **client-side cache / service-worker / self-update / version-visibility** class, which is a distinct and equally embarrassing failure mode for a framework that markets a "desktop-class app."

## The incident (generic)

A wesktop consumer app shipped a hand-rolled service worker that cached the app shell with **stale-while-revalidate** under a **fixed cache name** that never changed across builds. Result: users were trapped on a months-old cached app shell. Hard-refresh did not help (the SW intercepts navigations and serves the cached shell), so the user ran stale code with no indication anything was wrong — and no way to tell *which* build they were on. Clicking did nothing, a feature "didn't work at all," and every "just reload" instruction failed, because none of it was reaching the browser. For a desktop app this is catastrophic and invisible: the app silently refuses to update itself.

Every root cause below is something the **framework** should make impossible, because every consumer that hand-rolls caching will get it wrong the same way.

## The bug classes (with framework-level fixes)

### 1. Consumers hand-roll service workers (the footgun)
**Symptom:** a DIY `sw.js` with a fixed cache name + SWR-on-HTML traps users on stale shells; the framework offers nothing, so every consumer reinvents (and mis-implements) offline/cache.
**Fix (wesktop):** ship a **blessed, correct, opt-in service worker** generated by the framework, so consumers never write cache logic by hand. It must, by construction:
- Use a **per-build cache name** (busted on every build id) so `activate` purges all prior caches automatically.
- Be **network-first for navigations/HTML** (the shell is never served stale), **cache-first for content-hashed assets** (immutable), and **never cache** API/WS paths.
- `skipWaiting()` + `clients.claim()` + a **controllerchange → reload-once** flow so a new build self-applies with no user action.
Consumers opt in with a flag; they never author the SW.

### 2. No build/version identity — users can't see what they're running
**Symptom:** "which build am I on?" required reading source and diffing hashes; the user had to *ask for a version number to be added* mid-incident just to diagnose staleness.
**Fix (wesktop):** the framework **stamps a build id** (git hash / content hash / timestamp) into both the server and the client at build time, exposes it (`/__wesktop/version`, a `window.__WESKTOP__.buildId`, and a client helper), and provides a **drop-in version/build badge** component/util so "what version am I running" is always visible out of the box. Pair with the build-skew handshake from the companion todo so a stale client vs current backend is *detected*, not silently tolerated.

### 3. No self-update UX
**Symptom:** even after a correct build was deployed, nothing told the running app "an update is available" or applied it.
**Fix (wesktop):** built-in update detection (SW `updatefound`/`controllerchange`, or a version poll against `/__wesktop/version`) with a standard, overridable "updating…" reload — automatic in packaged/native mode, unobtrusive in browser mode.

### 4. Server-side cache headers for shell vs assets are not set correctly by default
**Symptom:** the shell (`index.html`) and `sw.js` could be cached; nothing enforced revalidation, so even absent a SW the browser could hold a stale shell.
**Fix (fastware/wesktop static+SPA serving):** by default serve `index.html` and `sw.js` with `Cache-Control: no-cache` (always revalidate), content-hashed `/assets/*` as `immutable, max-age=31536000`. Consumers get correct caching without thinking about it. (The build-id substitution into a public/`sw.js` — which bypasses the bundler's `define` — is another sharp edge the framework's SW generator should own end-to-end.)

### 5. Native/packaged mode ships a browser-cache footgun it doesn't need, and offers no reset
**Symptom:** a native webview window (local server, same-origin) gained nothing from an offline SW but inherited all its staleness risk, and there was **no in-app or CLI way to clear the webview/app cache** — the only escape was hand-unregistering the SW in DevTools, which a desktop-app user should never have to open.
**Fix (wesktop, `open`/native):**
- In native mode, **prefer no SW** (the server is local; offline caching is pointless) or force the always-fresh strategy.
- Provide a first-class **cache/state reset**: a `<app> reset-cache` CLI command and/or a Help-menu "Reset app data" that programmatically clears the webview's HTTP cache and unregisters service workers (pywebview/WebKit + platform webview APIs), plus clears wesktop-managed caches. On native launch, optionally verify the loaded build id matches the server's and auto-reset on mismatch.
- Expose a **"remove a service worker we regret shipping"** escape hatch in the framework: a generated self-unregistering `sw.js` mode (unregister + `caches.delete` all + reload clients) so a consumer that already shipped a bad SW can push a clean-up build without asking users to touch DevTools.

## The unifying fix

A desktop-app framework **owns the update-and-cache story**: correct-by-default caching, a visible build id, and automatic self-update — none of it hand-rolled by consumers. "The app won't update and there's no way to tell" must be structurally impossible, the same way the companion todo makes "the frontend talks to the wrong backend" impossible.

## Affected (framework side)

wesktop: a generated blessed service worker (+ self-unregister mode), build-id stamping + version endpoint + badge helper, self-update UX, native-mode cache/SW reset (CLI `reset-cache` + menu affordance + launch-time build-id check), docs steering consumers away from hand-rolled SWs. fastware: default `Cache-Control` for shell/`sw.js`/hashed assets in static+SPA serving; the `/__wesktop/version` (or `/__fastware/version`) endpoint; build-id injection that reaches `public/` assets like `sw.js`.

## Effort

M–L. Priority by embarrassment-per-effort: **#4 (S, correct default cache headers)** → **#2 (S–M, build-id stamp + version endpoint + badge)** → **#1 (M, blessed SW generator)** → **#5 (M, native reset + self-unregister escape hatch)** → **#3 (M, self-update UX)**.
